## 2.2 Root-Level Files

| File | Purpose |
|---|---|
| `docker-compose.yml` | Development-only orchestration file; spins up MySQL, Nginx, phpMyAdmin, and the backend together; not used in production |
| `.env` | Environment variables for all services (not committed to version control) |
| `.env.example` | Template listing all required environment variables with placeholder values |
| `.gitignore` | Excludes `.env`, `node_modules/`, build outputs, uploaded ZIPs, and project source directories |
| `package.json` | Root-level package for workspace scripts (e.g., dev start, seed commands) |
| `README.md` | Project documentation |

### Deploy Directory — `deploy/`

| File | Purpose |
|---|---|
| `deploy/nginx/00-upstream.conf` | Defines the `acadhost_backend` upstream pointing to the Express.js backend; loaded first alphabetically so upstream is available to all other config files |
| `deploy/nginx/acadhost.com.conf` | Platform root domain: serves student dashboard static files from `STUDENT_DASHBOARD_DIST`, proxies `/api/*` to Express.js backend, blocks internal phpMyAdmin verify endpoint |
| `deploy/nginx/admin.acadhost.com.conf` | Admin subdomain: serves admin dashboard static files from `ADMIN_DASHBOARD_DIST`, proxies `/api/*` to Express.js backend |
| `deploy/nginx/phpmyadmin.acadhost.com.conf` | phpMyAdmin subdomain: proxies to phpMyAdmin container |
| `deploy/nginx/default-catch-all.conf` | Default server block: returns 404 for unmatched subdomains |

These files are committed to version control and copied to `NGINX_CONF_DIR` during production deployment. They contain production-specific values. Development versions with Docker service names are maintained separately or generated during dev setup.

## 2.5 Nginx Configuration Directory — `nginx/`

| Path | Purpose |
|---|---|
| `nginx/conf.d/acadhost/` | Include directory for all Nginx config files; contains static platform configs (copied from `deploy/nginx/` during deployment) and per-project configs (generated by `nginxService.js` at runtime); all `*.conf` files in this directory are loaded by Nginx via `include /etc/nginx/conf.d/acadhost/*.conf;` |

`nginxService.js` writes one `.conf` file per active student project into this directory (e.g., `jane-portfolio.conf`, `bob-api.conf`). Static platform configs (`00-upstream.conf`, `acadhost.com.conf`, `admin.acadhost.com.conf`, `phpmyadmin.acadhost.com.conf`, `default-catch-all.conf`) are copied from `deploy/nginx/` during deployment and are NOT managed by `nginxService.js`.

The main `nginx.conf` (managed by the system, not by the application) must include:

```nginx
include /etc/nginx/conf.d/acadhost/*.conf;
```

In development, `docker-compose.yml` bind-mounts `./nginx/conf.d/acadhost/` into the Nginx container at `/etc/nginx/conf.d/acadhost/`.

## 2.6 Host-Level Project Directories (Runtime)

These directories exist on the host file system at runtime. They are not part of the repository. The base path is controlled by the `PROJECTS_BASE_DIR` environment variable.

| Environment | `PROJECTS_BASE_DIR` |
|---|---|
| Production | `/home/acadhost/projects` |
| Windows Development | `C:/acadhost/projects` |

### Per-Student Directory Structure

```
{PROJECTS_BASE_DIR}/
└── {student_id}/
    └── {project_id}/
        ├── source/
        │   ├── frontend/        (if frontend or combined project)
        │   └── backend/         (if backend or combined project)
        ├── build/
        │   └── logs/
        │       └── {build_timestamp}.log
        ├── uploads/
        └── Dockerfile
```

| Directory / File | Purpose |
|---|---|
| `{PROJECTS_BASE_DIR}/{student_id}/` | Top-level directory for all projects belonging to a specific student |
| `{PROJECTS_BASE_DIR}/{student_id}/{project_id}/` | Root directory for a specific project |
| `source/` | Extracted or cloned project source code |
| `source/frontend/` | Frontend source (present for frontend-only and combined projects) |
| `source/backend/` | Backend source (present for backend-only and combined projects) |
| `build/logs/` | Build log files; each build creates a timestamped log file; logs retained for 7 days |
| `uploads/` | Temporary storage for uploaded ZIP files; files are deleted immediately after successful extraction |
| `Dockerfile` | Generated Dockerfile for this project, copied from the appropriate template (`Dockerfile.node`, `Dockerfile.python`, `Dockerfile.frontend`, `Dockerfile.combined.node`, or `Dockerfile.combined.python`) and customized with the selected runtime version |

## 2.7 Nginx Configuration Directory (Runtime)

| Environment | Nginx Config Directory | Notes |
|---|---|---|
| Production | `/etc/nginx/conf.d/acadhost/` | Nginx runs natively; `nginxService.js` writes per-project files directly to this directory |
| Development | `./nginx/conf.d/acadhost/` (repo-relative, mounted into Docker container) | Nginx runs in Docker; directory is bind-mounted from the repo |

The actual path used by `nginxService.js` is configurable via the `NGINX_CONF_DIR` environment variable, consistent with the spec's mandate that all file system paths are configurable via environment variables.

## 2.8 Docker-Compose Development Configuration

```yaml
# docker-compose.yml (repo root — development only)
version: "3.8"

services:
  mysql:
    image: mysql:8.0
    container_name: acadhost-mysql
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
    ports:
      - "${MYSQL_PORT:-3306}:3306"
    volumes:
      - mysql_data:/var/lib/mysql

  phpmyadmin:
    image: phpmyadmin/phpmyadmin:latest
    container_name: acadhost-phpmyadmin
    restart: unless-stopped
    environment:
      PMA_HOST: mysql
      PMA_PORT: 3306
      PHPMYADMIN_BASE_PATH: ${PHPMYADMIN_BASE_PATH:-/phpmyadmin}
    ports:
      - "${PHPMYADMIN_PORT:-8080}:80"
    depends_on:
      - mysql

  nginx:
    image: nginx:latest
    container_name: acadhost-nginx
    restart: unless-stopped
    ports:
      - "${NGINX_PORT:-80}:80"
    volumes:
      - ./nginx/conf.d/acadhost:/etc/nginx/conf.d/acadhost
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro

  backend:
    build:
      context: ./backend
    container_name: acadhost-backend
    restart: unless-stopped
    env_file:
      - .env
    ports:
      - "${BACKEND_PORT:-3000}:3000"
    depends_on:
      - mysql
    volumes:
      - ./backend:/app
      - ${PROJECTS_BASE_DIR:-C:/acadhost/projects}:/projects

volumes:
  mysql_data:
```

## 2.9 Environment Variables for File Paths and Services

All file system paths used by the backend are defined via environment variables, as mandated by the spec.

### File Path Variables

| Environment Variable | Purpose | Production Default | Windows Dev Default |
|---|---|---|---|
| `PROJECTS_BASE_DIR` | Base directory for all student project files | `/home/acadhost/projects` | `C:/acadhost/projects` |
| `NGINX_CONF_DIR` | Directory where `nginxService.js` writes per-project Nginx config files; static platform configs are also placed here during deployment | `/etc/nginx/conf.d/acadhost` | `./nginx/conf.d/acadhost` |

### phpMyAdmin Variables

| Environment Variable | Purpose | Production Default | Windows Dev Default |
|---|---|---|---|
| `PHPMYADMIN_PORT` | Port on which phpMyAdmin is accessible on the host | `8080` | `8080` |
| `PHPMYADMIN_BASE_PATH` | Base URL path for phpMyAdmin | `/phpmyadmin` | `/phpmyadmin` |
| `PHPMYADMIN_URL` | Full base URL used by the backend to construct phpMyAdmin links for students | `http://localhost:8080` | `http://localhost:8080` |

### Database Credential Variables (Injected into Student Containers at Runtime)

These variables are not set in `.env`. They are injected by the backend into each student container at runtime during container creation via `dockerService.js`.

| Environment Variable | Purpose |
|---|---|
| `DB_HOST` | MySQL server hostname (always `host.docker.internal` for containers) |
| `DB_PORT` | MySQL server port |
| `DB_USER` | Database-specific restricted username |
| `DB_PASSWORD` | Database-specific restricted user password |
| `DB_NAME` | Database schema name |

## 2.10 Ambiguity Decisions Registry

All ambiguities encountered during Section 2 generation and their resolutions.

| # | Ambiguity | Decision | Rationale |
|---|---|---|---|
| 1 | Spec does not state whether Nginx config uses a single file or per-project snippets | Per-project `.conf` files in an include directory (`NGINX_CONF_DIR`); one file per student project generated by `nginxService.js`; static platform configs committed to `deploy/nginx/` and copied to the include directory during deployment | Per-project files allow O(1) add/remove operations; static platform configs are version-controlled and not managed by application code |
| 2 | Spec does not define where uploaded ZIP files are temporarily stored | `{PROJECTS_BASE_DIR}/{student_id}/{project_id}/uploads/`; deleted after extraction | Keeps temp files scoped to the student's project directory |
| 3 | Spec does not define the Nginx configuration directory on the production VM or the dev bind-mount strategy | Production: `/etc/nginx/conf.d/acadhost/`; Dev: `./nginx/conf.d/acadhost/` bind-mounted; path controlled by `NGINX_CONF_DIR` env var | Standard Nginx include pattern; env var satisfies the no-hardcoded-paths mandate |
| 4 | Spec does not provide exact `docker-compose.yml` contents | Provided concrete file with MySQL 8.0, Nginx latest, phpMyAdmin, backend build context, configurable ports, named volume | Standard choices consistent with the spec's technology stack |
| 5 | Spec does not name specific database credential environment variables injected into containers | Locked as `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME` | Industry-standard names; locked now to ensure cross-section consistency |
| 6 | Spec does not clarify if frontend-only projects run as containers or are served directly by host Nginx | Container with internal Nginx; host Nginx reverse-proxies to assigned port | Keeps all project types uniform — every project is a container; no special-casing in Nginx or build logic |
| 7 | Spec does not mention a Dockerfile for the Express.js API server itself (distinct from project templates) | `backend/Dockerfile` added for use by `docker-compose.yml` in development | `docker-compose.yml` references `build: context: ./backend` which requires this file |
| 8 | `Dockerfile.combined` did not account for two possible backend runtimes | Split into `Dockerfile.combined.node` and `Dockerfile.combined.python` | Combined projects can pair a frontend with either Node.js or Python; each needs a distinct template |
| 9 | Spec mentions phpMyAdmin but does not define its deployment method or configuration | Added as standalone container in `docker-compose.yml` for dev; env vars added for port, base path, and URL | phpMyAdmin is part of the tech stack and needs explicit deployment configuration |
