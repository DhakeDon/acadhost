# Section 8 — Nginx Reverse Proxy

## 8.1 Overview

Nginx serves as the reverse proxy and static file server for AcadHost. It handles four categories of traffic based on the `Host` header:

| Priority | Server Name | Routing Target | Config Source |
|---|---|---|---|
| 1 | `acadhost.com` | Student dashboard (static files) + `/api/*` proxy to Express.js backend | `deploy/nginx/acadhost.com.conf` (static, version-controlled) |
| 2 | `admin.acadhost.com` | Admin dashboard (static files) + `/api/*` proxy to Express.js backend | `deploy/nginx/admin.acadhost.com.conf` (static, version-controlled) |
| 3 | `phpmyadmin.acadhost.com` | phpMyAdmin container proxy | `deploy/nginx/phpmyadmin.acadhost.com.conf` (static, version-controlled) |
| 4 | `*.acadhost.com` | Student project containers on assigned ports | Per-project files generated by `nginxService.js` |
| — | `_` (catch-all) | `404 Not Found` | `deploy/nginx/default-catch-all.conf` (static, version-controlled) |

### Key Principles

| Principle | Detail |
|---|---|
| Plain HTTP only | Nginx receives traffic internally from the Cloudflare Tunnel; SSL is terminated at the Cloudflare edge; no port 443, no `certbot`, no Let's Encrypt on the VM |
| Include directory | All config files are placed in `NGINX_CONF_DIR`; Nginx loads them via `include /etc/nginx/conf.d/acadhost/*.conf;` |
| Static platform configs | The root domain, admin, phpMyAdmin, upstream, and catch-all configs are version-controlled files in `deploy/nginx/`; they are copied to `NGINX_CONF_DIR` during deployment and are NOT managed by `nginxService.js` |
| Per-project configs | `nginxService.js` writes one `.conf` file per active student project into `NGINX_CONF_DIR`; O(1) add/remove operations |
| Environment-variable-driven paths | `NGINX_CONF_DIR`, `STUDENT_DASHBOARD_DIST`, `ADMIN_DASHBOARD_DIST`, and `NGINX_PROXY_HOST` control all paths and targets |

### Nginx-Related Environment Variables (Locked in Section 3)

| Variable | Production Value | Development Value | Source Section |
|---|---|---|---|
| `NGINX_CONF_DIR` | `/etc/nginx/conf.d/acadhost` | `./nginx/conf.d/acadhost` | Section 3.2.4 |
| `NGINX_RELOAD_CMD` | `nginx -s reload` | `docker exec acadhost-nginx nginx -s reload` | Section 3.2.6 |
| `NGINX_TEST_CMD` | `nginx -t` | `docker exec acadhost-nginx nginx -t` | Section 3.2.6 |
| `NGINX_PROXY_HOST` | `127.0.0.1` | `host.docker.internal` | Section 3.2.5 |
| `BACKEND_PORT` | `3000` | `3000` | Section 3.2.1 |
| `PLATFORM_DOMAIN` | `acadhost.com` | `acadhost.com` | Section 3.2.11 |
| `CONTAINER_INTERNAL_PORT` | `8080` | `8080` | Section 3.2.5 |
| `STUDENT_DASHBOARD_DIST` | `/var/www/acadhost/student` | `../frontend/student-dashboard/dist` | Section 3.2.4 |
| `ADMIN_DASHBOARD_DIST` | `/var/www/acadhost/admin` | `../frontend/admin-dashboard/dist` | Section 3.2.4 |
| `PHPMYADMIN_PORT` | `8080` | `8080` | Section 3.2.8 |

## 8.2 Traffic Flow

### 8.2.1 Production Traffic Flow

```
Internet -> *.acadhost.com
    -> Cloudflare Edge (SSL termination, DNS resolution)
    -> cloudflared tunnel daemon on VM
    -> Nginx on VM (port 80, plain HTTP)
    -> routing decision based on Host header:
        |-- acadhost.com
        |   |-- /api/*  -> proxy_pass to Express.js backend (127.0.0.1:3000)
        |   +-- /*      -> static files from /var/www/acadhost/student
        |-- admin.acadhost.com
        |   |-- /api/*  -> proxy_pass to Express.js backend (127.0.0.1:3000)
        |   +-- /*      -> static files from /var/www/acadhost/admin
        |-- phpmyadmin.acadhost.com
        |   +-- /*      -> proxy_pass to phpMyAdmin (127.0.0.1:8080)
        |-- {subdomain}.acadhost.com
        |   +-- /*      -> proxy_pass to container (127.0.0.1:{containerPort})
        +-- (unmatched)
            +-- 404
```

### 8.2.2 Development Traffic Flow

```
Browser -> localhost:80 (mapped via docker-compose.yml)
    -> Nginx Docker container (acadhost-nginx)
    -> routing decision based on Host header:
        |-- acadhost.com (via /etc/hosts)
        |   |-- /api/*  -> proxy_pass to acadhost-backend:3000
        |   +-- /*      -> static files from STUDENT_DASHBOARD_DIST
        |-- admin.acadhost.com (via /etc/hosts)
        |   |-- /api/*  -> proxy_pass to acadhost-backend:3000
        |   +-- /*      -> static files from ADMIN_DASHBOARD_DIST
        |-- phpmyadmin.acadhost.com (via /etc/hosts)
        |   +-- /*      -> proxy_pass to acadhost-phpmyadmin:80
        |-- {subdomain}.acadhost.com (via /etc/hosts)
        |   +-- /*      -> proxy_pass to host.docker.internal:{containerPort}
        +-- (unmatched)
            +-- 404
```

In development, subdomain routing requires local `/etc/hosts` entries or can be skipped entirely during unit development (Section 1.6). Alternatively, developers access React dashboards directly via their Vite dev servers (e.g., `localhost:5173` for student, `localhost:5174` for admin) during active development, bypassing Nginx for frontend serving.

## 8.3 Nginx Deployment — Development vs. Production

| Aspect | Development | Production |
|---|---|---|
| Nginx instance | Docker container `acadhost-nginx` via `docker-compose.yml` | Runs natively on the Ubuntu VM |
| Config directory | `./nginx/conf.d/acadhost/` (bind-mounted into container) | `/etc/nginx/conf.d/acadhost/` |
| Reload command | `docker exec acadhost-nginx nginx -s reload` | `nginx -s reload` |
| Test command | `docker exec acadhost-nginx nginx -t` | `nginx -t` |
| Listening port | Port `80` inside container; mapped to `${NGINX_PORT:-80}` on host | Port `80` natively |
| SSL | None — development over plain HTTP | None — Cloudflare Tunnel provides SSL at the edge |
| Backend proxy_pass target | `acadhost-backend:3000` (Docker service name) | `127.0.0.1:3000` (localhost) |
| Student container proxy_pass target | `host.docker.internal:{containerPort}` | `127.0.0.1:{containerPort}` |
| phpMyAdmin proxy_pass target | `acadhost-phpmyadmin:80` (Docker service name; internal port) | `127.0.0.1:8080` (host-mapped port) |
| Dashboard static files | From `STUDENT_DASHBOARD_DIST` / `ADMIN_DASHBOARD_DIST` (or Vite dev server directly) | From `/var/www/acadhost/student` and `/var/www/acadhost/admin` |

## 8.4 Include Directory Architecture

### 8.4.1 Directory Structure

```
{NGINX_CONF_DIR}/
|-- 00-upstream.conf                    (static -- from deploy/nginx/)
|-- acadhost.com.conf                   (static -- from deploy/nginx/)
|-- admin.acadhost.com.conf             (static -- from deploy/nginx/)
|-- phpmyadmin.acadhost.com.conf        (static -- from deploy/nginx/)
|-- default-catch-all.conf              (static -- from deploy/nginx/)
|-- jane-portfolio.conf                 (generated by nginxService.js)
|-- bob-api.conf                        (generated by nginxService.js)
+-- alice-shop.conf                     (generated by nginxService.js)
```

### 8.4.2 File Ownership

| File(s) | Managed By | When Modified |
|---|---|---|
| `00-upstream.conf` | Manual deployment (copied from `deploy/nginx/`) | Only during initial deployment or infrastructure changes |
| `acadhost.com.conf` | Manual deployment (copied from `deploy/nginx/`) | Only during initial deployment or infrastructure changes |
| `admin.acadhost.com.conf` | Manual deployment (copied from `deploy/nginx/`) | Only during initial deployment or infrastructure changes |
| `phpmyadmin.acadhost.com.conf` | Manual deployment (copied from `deploy/nginx/`) | Only during initial deployment or infrastructure changes |
| `default-catch-all.conf` | Manual deployment (copied from `deploy/nginx/`) | Only during initial deployment or infrastructure changes |
| `{subdomain}.conf` | `nginxService.js` | Created on project deploy; deleted on project delete/terminate/student removal |

### 8.4.3 Nginx Main Configuration Requirement

The system Nginx main configuration (`/etc/nginx/nginx.conf`) must include the following directive inside the `http {}` block:

```nginx
include /etc/nginx/conf.d/acadhost/*.conf;
```

This loads all `.conf` files from the include directory. Files are loaded in alphabetical order; `00-upstream.conf` sorts first, ensuring the upstream definition is available to all subsequent server blocks.

## 8.5 Static Platform Config Files — `deploy/nginx/`

These files are committed to the repository under `deploy/nginx/` and copied to `NGINX_CONF_DIR` during deployment. They contain production values. For development, environment-specific versions are generated or different targets are used.

### 8.5.1 `deploy/nginx/00-upstream.conf`

```nginx
# AcadHost -- Backend Upstream Definition
# Static config -- committed to version control

upstream acadhost_backend {
    server 127.0.0.1:3000;
}
```

**Notes:**

- Defines the `acadhost_backend` upstream used by `acadhost.com.conf` and `admin.acadhost.com.conf`.
- The `00-` prefix ensures this file sorts first alphabetically, so the upstream is defined before any server blocks reference it.
- In development, the backend target is `acadhost-backend:3000` (Docker service name). The development version of this file must use that target.

### 8.5.2 `deploy/nginx/acadhost.com.conf`

```nginx
# AcadHost -- Student Dashboard + API
# Static config -- committed to version control

server {
    listen 80;
    server_name acadhost.com;

    # Maximum upload size (ZIP files up to 200 MB + 10 MB overhead)
    client_max_body_size 210m;

    # Block internal phpMyAdmin verify endpoint from external access
    location = /api/auth/phpmyadmin/verify {
        deny all;
        return 403;
    }

    # API proxy to Express.js backend
    location /api/ {
        proxy_pass http://acadhost_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";

        # SSE support -- disable buffering for streaming responses
        proxy_buffering off;
        proxy_cache off;
    }

    # Student dashboard -- static files
    location / {
        root /var/www/acadhost/student;
        try_files $uri $uri/ /index.html;
    }
}
```

**Notes:**

- `location = /api/auth/phpmyadmin/verify` uses exact match (`=`) for highest priority. It denies all external requests to this internal-only endpoint (Section 6.2.9).
- `location /api/` has higher priority than `location /` due to longer prefix match. All API requests are proxied to the Express.js backend.
- `location /` serves the student dashboard React build output. `try_files $uri $uri/ /index.html` supports SPA client-side routing.
- `proxy_buffering off` and `proxy_cache off` are required for SSE build log streaming (Section 6.5.7).
- `client_max_body_size 210m` allows ZIP uploads up to 200 MB + 10 MB multipart overhead.
- The `root` path (`/var/www/acadhost/student`) is the production value of `STUDENT_DASHBOARD_DIST`.

### 8.5.3 `deploy/nginx/admin.acadhost.com.conf`

```nginx
# AcadHost -- Admin Dashboard + API
# Static config -- committed to version control

server {
    listen 80;
    server_name admin.acadhost.com;

    # Allow Excel file uploads for student invitation (up to 5 MB + overhead)
    client_max_body_size 10m;

    # Block internal phpMyAdmin verify endpoint from external access
    location = /api/auth/phpmyadmin/verify {
        deny all;
        return 403;
    }

    # API proxy to Express.js backend
    location /api/ {
        proxy_pass http://acadhost_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";

        proxy_buffering off;
        proxy_cache off;
    }

    # Admin dashboard -- static files
    location / {
        root /var/www/acadhost/admin;
        try_files $uri $uri/ /index.html;
    }
}
```

**Notes:**

- The admin dashboard is served on the `admin` reserved subdomain (Section 1.11).
- The `/api/` proxy is required because the admin dashboard must call the Express.js backend for all operations. Without this proxy, the admin dashboard would need cross-origin requests to `acadhost.com/api/*`, which would break `SameSite=Strict` refresh token cookies. By proxying `/api/*` through the admin subdomain, all API calls are same-origin.
- `client_max_body_size 10m` accommodates Excel file uploads for student invitation (5 MB limit + overhead). Nginx default of `1m` would block these uploads.

### 8.5.4 `deploy/nginx/phpmyadmin.acadhost.com.conf`

```nginx
# AcadHost -- phpMyAdmin
# Static config -- committed to version control

server {
    listen 80;
    server_name phpmyadmin.acadhost.com;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

**Notes:**

- `phpmyadmin` is a reserved subdomain (Section 1.11) -- students cannot claim it.
- The `proxy_pass` target is `127.0.0.1:8080` in production (`PHPMYADMIN_PORT`). In development, this is `acadhost-phpmyadmin:80` (Docker service name; phpMyAdmin internal port is `80`, host-mapped port is `8080`).

### 8.5.5 `deploy/nginx/default-catch-all.conf`

```nginx
# AcadHost -- Default Catch-All
# Static config -- committed to version control

server {
    listen 80 default_server;
    server_name _;

    return 404;
}
```

**Notes:**

- Catches any request whose `Host` header does not match any defined `server_name`.
- Returns `404 Not Found` for unregistered subdomains, reserved subdomains without routing (e.g., `api.acadhost.com`, `www.acadhost.com`), and direct IP access.

## 8.6 Per-Project Config Files — `nginxService.js`

`nginxService.js` (Section 2.3) generates and manages one `.conf` file per active student project in the `NGINX_CONF_DIR` directory.

### 8.6.1 Functions

| Function | Description |
|---|---|
| `addProjectConfig(subdomain, containerPort)` | Writes `{NGINX_CONF_DIR}/{subdomain}.conf` with the project server block; validates config via `NGINX_TEST_CMD`; reloads Nginx via `NGINX_RELOAD_CMD` |
| `removeProjectConfig(subdomain)` | Deletes `{NGINX_CONF_DIR}/{subdomain}.conf`; reloads Nginx |
| `removeMultipleProjectConfigs(subdomains)` | Deletes multiple `.conf` files; reloads Nginx once after all deletions |
| `initializeOnStartup()` | On backend startup: queries all active projects and ensures a `.conf` file exists for each; removes orphaned `.conf` files for projects that no longer exist; reloads Nginx once |

### 8.6.2 Per-Project Config Template

```nginx
# Auto-generated by nginxService.js -- DO NOT EDIT
# Project: {subdomain} | Port: {containerPort} | Generated: {ISO_TIMESTAMP}

server {
    listen 80;
    server_name {subdomain}.{PLATFORM_DOMAIN};

    location / {
        proxy_pass http://{NGINX_PROXY_HOST}:{containerPort};
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

### 8.6.3 Template Variables

| Placeholder | Value | Source |
|---|---|---|
| `{subdomain}` | `projects.subdomain` | Section 4.2.2 |
| `{containerPort}` | `projects.container_port` | Section 4.2.2 |
| `{PLATFORM_DOMAIN}` | `process.env.PLATFORM_DOMAIN` (default `acadhost.com`) | Section 3.2.11 |
| `{NGINX_PROXY_HOST}` | `process.env.NGINX_PROXY_HOST` (production: `127.0.0.1`, dev: `host.docker.internal`) | Section 3.2.5 |
| `{ISO_TIMESTAMP}` | Current ISO 8601 timestamp | `new Date().toISOString()` |

### 8.6.5 When `nginxService.js` Is Called

| Trigger | Function Called | Action |
|---|---|---|
| Successful project deployment | `addProjectConfig(subdomain, port)` | Write `{subdomain}.conf`; validate; reload |
| Project delete by student | `removeProjectConfig(subdomain)` | Delete `{subdomain}.conf`; reload |
| Project terminate by admin | `removeProjectConfig(subdomain)` | Delete `{subdomain}.conf`; reload |
| Student removal (individual) | `removeMultipleProjectConfigs(subdomains)` | Delete all project `.conf` files; reload once |
| Student removal (batch) | `removeMultipleProjectConfigs(subdomains)` | Delete all affected `.conf` files; reload once |
| Backend server startup | `initializeOnStartup()` | Sync `.conf` files with database; reload once |
| Webhook rebuild | NOT called | Same port, same subdomain; no config change |
| Database switch | NOT called | Same port, same subdomain |
| Resource update fallback | NOT called | Same port, same subdomain |
| Project stop | NOT called | Config retained; Nginx returns 502 |
| Project restart | NOT called | Config already exists |

### 8.6.6 Write and Reload Algorithm

```
function addProjectConfig(subdomain, containerPort):
  1. Generate config content from the per-project template (Section 8.6.2).
  2. Write to {NGINX_CONF_DIR}/{subdomain}.conf atomically:
     a. Write to {NGINX_CONF_DIR}/{subdomain}.conf.tmp
     b. Rename to {NGINX_CONF_DIR}/{subdomain}.conf
  3. Validate via NGINX_TEST_CMD (e.g., nginx -t):
     a. If validation succeeds (exit code 0): execute NGINX_RELOAD_CMD.
     b. If validation fails (exit code non-zero):
        - Log the error with full nginx -t output.
        - Delete the newly written {subdomain}.conf.
        - Throw an error to the calling function.
  4. Return success.

function removeProjectConfig(subdomain):
  1. Delete {NGINX_CONF_DIR}/{subdomain}.conf.
     If file does not exist, log a warning and continue.
  2. Execute NGINX_RELOAD_CMD.

function removeMultipleProjectConfigs(subdomains):
  1. For each subdomain: delete {NGINX_CONF_DIR}/{subdomain}.conf.
  2. Execute NGINX_RELOAD_CMD once after all deletions.

function initializeOnStartup():
  1. Query all projects WHERE status IN ('running', 'stopped', 'building', 'failed')
     AND container_port IS NOT NULL
     AND subdomain NOT LIKE '_deleted_%'.
  2. For each project: if {NGINX_CONF_DIR}/{subdomain}.conf does not exist,
     write it.
  3. List all .conf files in NGINX_CONF_DIR that are NOT in the set of
     expected filenames (static platform configs + active project subdomains).
     Delete orphaned project .conf files. Do not delete platform configs.
  4. Execute NGINX_TEST_CMD and NGINX_RELOAD_CMD once.
```

## 8.7 Proxy Header Details

Every `proxy_pass` location block includes the following headers:

| Header | Directive | Purpose |
|---|---|---|
| `Host` | `proxy_set_header Host $host;` | Preserves the original Host header so the backend or student application knows which domain was requested |
| `X-Real-IP` | `proxy_set_header X-Real-IP $remote_addr;` | Passes the connecting client IP; in production this is the cloudflared loopback address; real client IP is in CF-Connecting-IP (set by Cloudflare, not Nginx) |
| `X-Forwarded-For` | `proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;` | Appends client IP to the X-Forwarded-For chain for proxy-aware applications |
| `X-Forwarded-Proto` | `proxy_set_header X-Forwarded-Proto $scheme;` | Indicates the original protocol; $scheme is http on VM; real protocol (https) is set by Cloudflare |
| `Upgrade` | `proxy_set_header Upgrade $http_upgrade;` | Required for WebSocket support in student projects and backend |
| `Connection` | `proxy_set_header Connection "upgrade";` | Paired with Upgrade for WebSocket connection upgrading |

## 8.8 SSE (Server-Sent Events) Support

The build log streaming endpoint (`GET /api/projects/:id/build-logs/stream`, Section 6.5.7) uses Server-Sent Events. SSE requires that Nginx does not buffer the response.

### Nginx-Level Configuration

The `acadhost.com` and `admin.acadhost.com` server blocks include in their `/api/` location:

```nginx
proxy_buffering off;
proxy_cache off;
```

### Application-Level Header

The Express.js backend also sets `X-Accel-Buffering: no` on SSE responses (Section 6.5.7). Both the Nginx-level directives and the application-level header are set for defense-in-depth.

## 8.9 Upload Size Limits

| Server Block | `client_max_body_size` | Reason |
|---|---|---|
| `acadhost.com` | `210m` | ZIP file uploads up to 200 MB + 10 MB multipart overhead |
| `admin.acadhost.com` | `10m` | Excel file uploads for student invitation up to 5 MB + overhead |
| `phpmyadmin.acadhost.com` | Not set (Nginx default `1m`) | phpMyAdmin requests are small |
| Per-project `{subdomain}.conf` | Not set (Nginx default `1m`) | Student containers handle their own request sizes |
| `default-catch-all.conf` | Not set | Returns 404 immediately; no body processing |

## 8.10 Stopped Project Behavior

When a project is stopped, its per-project `.conf` file remains in `NGINX_CONF_DIR`. Requests to the stopped project subdomain reach Nginx, which attempts to proxy_pass to the container port. Since the container is not running, the connection is refused and Nginx returns a `502 Bad Gateway` response. This is intentional -- it clearly indicates the project is offline. The config file is retained so the subdomain remains claimed.

---

## 8.11 WebSocket Support

The `Upgrade` and `Connection` proxy headers are included in all `proxy_pass` location blocks (platform API, admin API, and student project server blocks). If a student project uses WebSockets (socket.io, native ws), the headers ensure the upgrade handshake is forwarded correctly.

## 8.12 phpMyAdmin Verify Endpoint — Nginx Block

The internal endpoint `POST /api/auth/phpmyadmin/verify` (Section 6.2.9) is blocked from external access by Nginx. Both `acadhost.com` and `admin.acadhost.com` server blocks include:

```nginx
location = /api/auth/phpmyadmin/verify {
    deny all;
    return 403;
}
```

| Detail | Value |
|---|---|
| Match type | Exact match (`=`) -- highest priority; checked before `location /api/` |
| Effect | Returns `403 Forbidden` for any external request to this path |
| Internal access | phpMyAdmin signon.php calls this endpoint directly via `host.docker.internal:{BACKEND_PORT}/api/auth/phpmyadmin/verify`, bypassing Nginx entirely |
| Student project subdomains | Not affected -- student server blocks proxy to containers, not to the backend |

## 8.13 Cloudflare Tunnel Integration

| Parameter | Value |
|---|---|
| Tunnel daemon | `cloudflared` installed on the VM |
| Tunnel target | `http://localhost:80` (Nginx) |
| DNS | `*.acadhost.com` wildcard DNS managed by Cloudflare; points to the tunnel |
| SSL | Terminated at the Cloudflare edge; cloudflared forwards plain HTTP to Nginx |
| Client IP | Real client IP in `CF-Connecting-IP` header (set by Cloudflare); Nginx `$remote_addr` is cloudflared loopback address |

### Cloudflare Tunnel Configuration

```yaml
# /etc/cloudflared/config.yml (production)
tunnel: <TUNNEL_ID>
credentials-file: /etc/cloudflared/.credentials.json

ingress:
  - hostname: "*.acadhost.com"
    service: http://localhost:80
  - hostname: "acadhost.com"
    service: http://localhost:80
  - service: http_status:404
```

This configuration is only used in production. cloudflared is not used during development (Section 1.6).

## 8.15 Configuration Validation

Before reloading Nginx, `nginxService.js` validates the configuration:

```
function validateAndReload():
  1. Execute NGINX_TEST_CMD (e.g., nginx -t) to test all loaded configs.
  2. If exit code 0 (success):
     - Execute NGINX_RELOAD_CMD to reload.
  3. If exit code non-zero (failure):
     - Log the error with the full test command output.
     - If triggered by addProjectConfig():
       - Delete the newly written {subdomain}.conf.
       - Throw an error to the calling function.
     - If triggered by removeProjectConfig():
       - Log but do not throw -- the file is already deleted;
         the next reload attempt will succeed.
```

## 8.16 Error Scenarios

| Scenario | Nginx Behavior | Resolution |
|---|---|---|
| Student container stopped | `502 Bad Gateway` | Expected; student/admin must restart |
| Student container crashed | Brief `502`; Docker `--restart unless-stopped` auto-recovers | Automatic |
| `NGINX_TEST_CMD` fails after new config | New `.conf` deleted; Nginx continues with previous valid config | Error logged; deployment reports failure |
| `NGINX_CONF_DIR` not writable | `nginxService.js` throws error; deployment fails | Check file permissions |
| Orphaned `.conf` file | Nginx routes to non-existent port; returns `502` | `initializeOnStartup()` cleans orphans on next restart |
| Platform static config syntax error | Nginx refuses to start | Fix config in `deploy/nginx/` and redeploy |

## 8.17 Same-Origin Authentication Architecture

Both dashboards proxy `/api/*` to the Express.js backend through their own Nginx server blocks. This ensures all API calls are same-origin:

| Dashboard | Browser Origin | API Call Target | Cookie Domain |
|---|---|---|---|
| Student | `https://acadhost.com` | `https://acadhost.com/api/*` | `acadhost.com` |
| Admin | `https://admin.acadhost.com` | `https://admin.acadhost.com/api/*` | `admin.acadhost.com` |

`SameSite=Strict` refresh token cookies work correctly because each dashboard and its API are on the same origin. No cross-origin requests are needed for authentication. Section 5.10 cookie configuration remains unchanged.

## 8.18 Development-Specific Notes

### 8.18.1 Development Platform Configs

For development, the static platform configs must use Docker service names instead of `127.0.0.1`. Development versions are needed for each file:

**Development `00-upstream.conf`:**

```nginx
upstream acadhost_backend {
    server acadhost-backend:3000;
}
```

**Development `phpmyadmin.acadhost.com.conf` (key difference):**

```nginx
server {
    listen 80;
    server_name phpmyadmin.acadhost.com;

    location / {
        proxy_pass http://acadhost-phpmyadmin:80;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

Development per-project configs use `NGINX_PROXY_HOST` (`host.docker.internal`) instead of `127.0.0.1`. This is handled automatically by `nginxService.js` reading `process.env.NGINX_PROXY_HOST`.

The `acadhost.com.conf` and `admin.acadhost.com.conf` development versions differ only in that they use the `acadhost_backend` upstream (which points to `acadhost-backend:3000` in dev) and development static file paths from `STUDENT_DASHBOARD_DIST` and `ADMIN_DASHBOARD_DIST`.

### 8.18.2 Subdomain Simulation in Development

| Approach | How | Trade-off |
|---|---|---|
| `/etc/hosts` entries | Add entries like `127.0.0.1 acadhost.com admin.acadhost.com phpmyadmin.acadhost.com jane-portfolio.acadhost.com` | Manual; requires per-project entries |
| Skip subdomain routing | Access backend directly via `localhost:3000`, dashboards via Vite dev servers (`localhost:5173`, `localhost:5174`), student containers via `localhost:{containerPort}` | Adequate for unit development; does not test Nginx routing |

## 8.19 Ambiguity Decisions Registry

| # | Ambiguity | Decision | Rationale |
|---|---|---|---|
| 1 | Spec does not define whether stopped projects show a custom error page or default 502 | Default Nginx 502 page | Custom error pages add complexity not in spec |
| 2 | Spec does not explicitly mention WebSocket support for student projects | Include Upgrade and Connection proxy headers in all server blocks | Standard practice; many student apps use WebSockets; zero cost |
| 3 | Spec does not provide cloudflared configuration file contents | Provided minimal config routing *.acadhost.com and acadhost.com to localhost:80 | Required for production; follows standard cloudflared ingress rules |
| 4 | Spec does not define whether nginxService.js validates config before reload | Always run NGINX_TEST_CMD before NGINX_RELOAD_CMD; delete invalid config on failure | Prevents silent config corruption |
| 5 | Spec does not define client_max_body_size values | 210m for root domain (ZIP uploads); 10m for admin (Excel uploads); default for others | Without explicit limits, Nginx rejects uploads over 1 MB |
| 6 | Spec does not define how admin dashboard accesses API without cross-origin issues | admin.acadhost.com server block includes /api/ proxy to backend | Keeps all API calls same-origin; SameSite=Strict cookies work |
| 7 | Spec does not define catch-all behavior for unmatched subdomains | default_server block returns 404 | Clean rejection; prevents routing to a random upstream |
| 8 | Spec does not define how platform Nginx configs are managed vs project configs | Static platform configs in deploy/nginx/ (version-controlled); per-project configs generated by nginxService.js | Separation of concerns; platform infra is not dynamically generated |
| 9 | Admin server block client_max_body_size not in spec | Added 10m to accommodate Excel file uploads (5 MB limit + overhead) | Nginx default 1 MB would block Excel uploads |
